# NSKK ユーザビリティ目標・評価計画

## 目的

本ドキュメントはNSKKのユーザビリティ評価に関する目標とテスト計画を定義します。リリース後にユーザビリティテストを実施するための枠組みを事前に設計しておくものです。

## 評価フレームワーク

### 使用する指標

| 指標 | 説明 | 目標値 |
|------|------|--------|
| **SUS** (System Usability Scale) | 10項目の標準ユーザビリティ尺度 | > 80/100 |
| **NPS** (Net Promoter Score) | 推奨意向スコア | > 50 |
| **タスク完了率** | 定義タスクの成功率 | > 95% |
| **エラー率** | タスク実行中のエラー発生率 | < 5% |
| **総合満足度** | ユーザー満足度 | > 95% |

### 評価方法論

テスト実施時には以下の方法を組み合わせる:

1. **タスクベーステスト**: 定義されたシナリオの実行と測定
2. **SUS/NPS調査**: 標準化されたアンケート
3. **観察・インタビュー**: 定性的データ収集
4. **A/Bテスト**: 設計選択の定量比較

## 計画テストシナリオ

### シナリオ1: 初回セットアップ

**タスク**:
1. NSKKのインストール
2. 基本設定の実施
3. 辞書の選択とロード
4. 最初の日本語入力テスト

**評価指標**:
- 完了時間（目標: < 5分）
- 成功率
- ヘルプ参照回数
- 満足度

### シナリオ2: 基本日本語入力

**タスク**:
1. ひらがな入力
2. カタカナ変換
3. 漢字変換
4. 送り仮名処理
5. 候補選択

**評価指標**:
- タスク完了率
- エラー率
- 完了時間
- 入力速度（文字/分）

### シナリオ3: 高度な機能

**タスク**:
1. 注釈機能の利用
2. 補完機能の活用
3. 学習機能の確認
4. カスタム辞書の追加
5. 入力方式の切り替え

**評価指標**:
- 完了率
- 満足度

### シナリオ4: 辞書カスタマイズ

**タスク**:
1. 個人辞書への単語登録
2. 辞書の優先順位変更
3. 辞書ファイルの直接編集
4. 辞書のインポート/エクスポート

**評価指標**:
- 完了率
- 満足度
- ドキュメント品質評価

### シナリオ5: パフォーマンス体感評価

**タスク**:
1. 長文入力（500文字以上）
2. 大規模辞書での検索
3. 連続入力テスト

**評価指標**:
- 応答性の体感評価
- 安定性の体感評価

## A/Bテスト計画

### テストA: インライン vs ポップアップ候補表示

**目的**: どちらの表示方式がユーザーに好まれるか、タスク効率に差があるかを比較する。

**評価指標**: タスク完了時間、エラー率、好評率

### テストB: 自動学習 ON vs OFF

**目的**: 自動学習がユーザー体験に与える影響を測定する。

**評価指標**: 変換精度、入力速度、満足度

### テストC: AZIK vs ローマ字入力

**目的**: AZIK経験者における入力方式の比較。

**評価指標**: 入力速度、エラー率、満足度

## SUS評価項目

テスト実施時に使用する標準SUS質問10項目:

1. このシステムを頻繁に使いたいと思う
2. システムは不必要に複雑だと思う
3. システムは使いやすいと思う
4. このシステムを使うには技術者の支援が必要だと思う
5. システムの様々な機能はうまく統合されていると思う
6. システムには矛盾が多いと思う
7. ほとんどの人はこのシステムを素早く使いこなせると思う
8. システムは非常に使いにくいと思う
9. システムを使うことに自信が持てた
10. システムを使う前に多くのことを学ぶ必要があった

(各項目5段階評価、SUSスコア算出式に従い100点満点に変換)

## 参加者プロファイル計画

| カテゴリ | 目標割合 | 特性 |
|---------|----------|------|

## 想定される改善領域

テスト前の設計段階で予測される改善ポイント:

### 初回セットアップ体験

**予想される課題**:
- 辞書選択の基準が初心者に不明確
- 設定項目が多すぎる可能性

**対策候補**:
- セットアップウィザードの提供
- 推奨設定プリセットの提供
- インタラクティブチュートリアル

### ドキュメント

**予想される課題**:
- 高度な機能のドキュメント不足
- 実例が少ない可能性

**対策候補**:
- 実践例の充実
- FAQ セクションの整備

### 移行ガイド

**対策候補**:
- ddskk → NSKK 移行ガイド
- skkeleton → NSKK 移行ガイド
- 設定変換ツールの提供

### エラーメッセージ

**予想される課題**:
- エラーメッセージが技術的すぎる可能性
- 解決方法の提示が不十分

**対策候補**:
- ユーザーフレンドリーなエラーメッセージ
- エラー時の自動リカバリー機能
- トラブルシューティングガイド

## 評価実施条件

ユーザビリティテストを実施するための前提条件:

1. 基本機能（ひらがな入力、漢字変換、候補選択）が動作すること
2. 辞書読み込みが機能すること
3. インストール手順が確立していること
4. テスト環境: Emacs 30+
