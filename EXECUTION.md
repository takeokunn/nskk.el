# 実行計画書

## 1. 要約
- 7層アーキテクチャを標榜しているが、各レイヤーの実装がスタブ状態で設計と乖離している。
- 特に `nskk-layer-core.el`, `nskk-layer-infrastructure.el`, `nskk-layer-extension.el` の未実装／不完全実装が上位レイヤー全体の破綻を引き起こしている。
- 実装済みの既存モジュール (`nskk-converter.el`, `nskk-thread-pool.el`, `nskk-dict-*` 等) との責務重複が多く、二重管理が発生している。

## 2. 現状診断
- `nskk-layer-core.el` : 変換エンジンの核となる関数が未実装スタブであり、`(cons input "")` を返すなど期待を満たしていない。半角／全角変換もオフセット計算がゼロで無効化されている。
- `nskk-layer-infrastructure.el` : スレッドプールや非同期処理の抽象を用意しているが、`nskk-infrastructure-submit-task` が同期待機で即座にコールバックするだけで、設計意図と乖離。リソース監視／メモリ管理も実働していない。
- `nskk-layer-extension.el` : フック登録で `add-hook` に適切なシンボルを渡さず、拡張ポイントが実質無効。イベント／メッセージングの抽象も最低限のバリデーションや例外処理が欠落。
- 既存モジュール群 (`nskk-converter.el`, `nskk-thread-pool.el`, `nskk-data*`, `nskk-extension*`) からの再実装が乱立し、API が二種類存在する。利用側はどちらを参照すべきか不明瞭で、保守コストが急増。

## 3. リファクタリング方針
1. **責務マッピングの再整理**
   - 既存の安定モジュールと 7 層構造の対応表を作成し、重複する抽象を廃止または統合する。
   - レイヤー毎の公開 API を最小限に定義し、上位レイヤーがどの関数を呼ぶのか明文化する。
2. **スタブの除去と実装統合**
   - `nskk-layer-core.el` から直接 `nskk-converter.el` / `nskk-search.el` 等を委譲するか、逆に既存実装をレイヤー側へ移設し単一実装に統一する。
   - 半角／全角変換など基礎ユーティリティは既存実装を再利用し、テストで挙動を保証する。
3. **インフラ層の現実的な抽象化**
   - 本当に非同期基盤が必要なら `nskk-thread-pool.el` を内包し、`nskk-infrastructure-submit-task` から委譲する。
   - シンプルな同期モデルで良いなら抽象レイヤー自体を薄くして、機能を限定する。
4. **拡張レイヤーの再設計**
   - Emacs 標準の hook 仕組みと互換な形でデータ構造を見直し、`add-hook` に有効なシンボルを渡すよう修正。
   - イベント履歴やメッセージングに型チェックとエラーハンドリングを追加し、公開仕様をドキュメント化。
5. **ドキュメント／テストの整備**
   - 各レイヤーの役割と利用例を Diátaxis に沿って再記述し、API の二重化を解消。
   - クリティカルパス（ローマ字変換、辞書検索、非同期タスク）の動作確認用テストを追加。

## 4. リスクと留意事項
- 二重実装を解消する際に既存コードを大きく移動する可能性が高く、API の互換性を確認する必要がある。
- 非同期機能を本格導入する場合、Emacs 31 以降のみ対応となるため互換性ポリシーの明文化が必須。
- フック／イベントの仕様変更は外部拡張に影響するため、段階的な移行策と告知が必要。

## 5. 推奨次アクション
1. レイヤー毎の責務表と依存関係図を整理した設計レビュー資料を作成。
2. `nskk-layer-core.el` のスタブを既存変換ロジックに差し替え、基本テストを通す。
3. インフラ層と拡張層の API を実働させるか、不要であれば段階的に縮退させる。
